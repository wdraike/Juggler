    Create the migration file. The `up` function must execute these steps in order inside a single transaction (knex.transaction):

    STEP 1 — Backfill (run while old ENUM values are still valid):
    Execute a raw UPDATE with this exact CASE expression against task_masters:
      WHEN placement_mode = 'marker'          THEN 'reminder'
      WHEN `when` LIKE '%allday%'              THEN 'all_day'
      WHEN `when` LIKE '%fixed%'              THEN 'fixed'
      WHEN preferred_time_mins IS NOT NULL     THEN 'time_window'
      WHEN `when` IS NOT NULL AND `when` != '' THEN 'time_blocks'
      ELSE                                         'anytime'
    The CASE is evaluated in order — marker check first, so existing marker rows
    map to reminder before the when-content checks fire. The `marker` column was
    dropped in migration 20260501000300, so check placement_mode = 'marker', not
    the column. All 7 old values must be covered: marker→reminder, fixed stays
    fixed, pinned_date→anytime (falls to ELSE), recurring_rigid→time_window
    (has preferredTimeMins), recurring_window→time_window, recurring_flexible→
    time_blocks or anytime depending on `when`, flexible→anytime.

    STEP 2 — MODIFY COLUMN to new ENUM:
    Use knex.raw to execute:
      ALTER TABLE task_masters MODIFY COLUMN placement_mode
        ENUM('reminder','all_day','fixed','time_window','time_blocks','anytime')
        NOT NULL DEFAULT 'anytime'
    (Default changes from 'flexible' to 'anytime'. The column currently has NOT NULL
    and a default, so MODIFY COLUMN preserves both if specified explicitly.)

    STEP 3 — Strip 'allday' and 'fixed' tokens from the `when` column:
    Try REGEXP_REPLACE first (available MySQL 8.0.4+):
      UPDATE task_masters
        SET `when` = TRIM(BOTH ',' FROM REGEXP_REPLACE(
          REPLACE(REPLACE(`when`, 'allday', ''), 'fixed', ''), ',+', ','))
        WHERE `when` LIKE '%allday%' OR `when` LIKE '%fixed%'
    Wrap this in a try/catch. If it throws (MySQL < 8.0.4), fall back to a JS loop:
    fetch all rows WHERE `when` LIKE '%allday%' OR `when` LIKE '%fixed%', strip
    both tokens in JS (split on comma, filter out 'allday' and 'fixed', rejoin),
    then batch-update using knex('task_masters').where('id', row.id).update({when: cleaned}).
    After stripping, set any empty-string `when` values to NULL:
      UPDATE task_masters SET `when` = NULL WHERE `when` = ''

    STEP 4 — Rebuild views:
    Drop both views then recreate them with the updated CASE expressions:
      DROP VIEW IF EXISTS tasks_with_sync_v
      DROP VIEW IF EXISTS tasks_v
    Recreate tasks_v using the exact SELECT structure from migration 20260501000300,
    with these two CASE column replacements:
      Line computing `rigid`:
        CASE WHEN m.placement_mode = 'fixed' THEN 1 ELSE 0 END AS rigid
        (was: CASE WHEN m.placement_mode = 'recurring_rigid' THEN 1 ELSE 0 END)
      Line computing `marker`:
        CASE WHEN m.placement_mode = 'reminder' THEN 1 ELSE 0 END AS marker
        (was: CASE WHEN m.placement_mode = 'marker' THEN 1 ELSE 0 END)
    Both instances of these CASE expressions appear in tasks_v (once in the recurring
    template SELECT, once in the instances SELECT — update both).
    Recreate tasks_with_sync_v using the exact SQL from 20260501000300 — it selects
    from tasks_v, so no enum references inside it; no changes needed.

    The `down` function should throw with message:
    'Down migration for placement_mode_enum_redesign not implemented — reverting
     requires reconstructing all old enum values from context that no longer exists.'
