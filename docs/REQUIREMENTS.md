### R50 — Overdue / Past-Due Item Handling

> **Model (David directive, 2026-06-21 — refines the initial framing).** An instance's "due date/time" is the spine of this requirement:
> - **Hard due date/time** — the user set a deadline, OR it is an **ingested calendar** date/time (fixed event). Not every task has one.
> - **Implied due date/time** — derived for **recurring instances** and **split chunks of a recurring task** (from the recurrence/occurrence boundary), since the user never typed one.
> - **No due date/time** — **nonrecurring** tasks (including nonrecurring split chunks) get one ONLY if the user set a deadline; otherwise they have none.
>
> **Three states** (drive scheduler + calendar): (1) **Overdue** — has a due date/time (hard or implied) that has PASSED and is incomplete (`status=''`) → the scheduler STOPS rescheduling it, it is PINNED at its due date/time, the calendar shows it there flagged overdue, and it appears in the dedicated "Past due" list. (2) **Unscheduled** — NOT past its due date/time (or has no due date) and the scheduler could not place it this run → unscheduled lane (keeps being retried on future runs). (3) **Placed** — not past due and placeable → normal.
>
> **Reconciliation with R32.7 / R32.4:** R32.7's "past-date occurrence left unplaced" governs recurring occurrences that never received an *implied* due/slot; once R50 computes an implied due for every recurring instance, an occurrence past its implied due becomes **Overdue** (not unplaced). R50's "Overdue" SUBSUMES the existing auto-`missed` mechanism (R32.4) — implement by extending it, not as a parallel state. ## Use cases

Three primary use cases are recorded in the Scooter KG (`has_use_case` facts) and corroborated by `juggler/docs/PROJECT-BRIEF.md §Users`.

### UC-1 — Individual contributor: track tasks and plan sprints

**Actor:** Developer or knowledge worker managing their own workload.

**Goal:** Schedule personal tasks efficiently across working days, track progress, and avoid overcommitment.

**Flow:**
1. User creates tasks (R1.1–R1.10) with duration estimates, priorities, and optional deadlines.
2. User organizes tasks into projects (R4.1–R4.5), optionally color-coded (R5.1).
3. User runs the scheduler (R11.1–R11.22) — tasks are placed in optimal slots respecting constraints.
4. User views the calendar grid (R8.1–R8.8) to see today's plan alongside calendar events.
5. User marks tasks `wip` as they start work, updating `time_remaining` (R6.1–R6.6).
6. User marks tasks `done` via status update (R2.1–R2.8).
7. User can invoke AI natural-language commands (R15.1–R15.5) to bulk-create or reschedule tasks.
8. User configures recurring tasks (R18.1–R18.8), splits (R19.1–R19.7), locations (R20.1–R20.4), and schedule templates (R21.1–R21.3) to fine-tune scheduling.

**MCP variant:** Same flow via `create_task` / `run_schedule` / `set_task_status` MCP tools (R17.1–R17.2).

### UC-2 — Team lead: view workload and monitor progress

**Actor:** Engineering manager or team lead overseeing a small team.

**Goal:** See the overall task backlog, understand scheduling status, and identify blockers.

**Flow:**
1. Lead uses the MCP interface (R17.1–R17.2) or the UI to query tasks by project or status.
2. Lead reviews the dependency graph (R10.1–R10.5) to identify blocked chains.
3. Lead reassigns tasks via task update (R2.1–R2.8), adjusting priorities and deadlines.
4. Lead uses calendar sync (R7.1–R7.8) to ensure team events are reflected in the schedule.

**Gap:** Dedicated team-workload view and assignment is not implemented as of 2026-06-15; the above flow is achievable via individual-user flows only.

### UC-3 — Freelancer: billable hours and project estimation

**Actor:** Independent contractor billing clients per hour or per project.

**Goal:** Track time spent on client projects, estimate future work, and produce billing data.

**Flow:**
1. Freelancer creates tasks (R1.1–R1.10) per client project with duration estimates.
2. Freelancer marks tasks as `wip` (R6.1–R6.6) to record time-in-progress against estimates.
3. Freelancer exports data (R22.1–R22.5) for external billing use.
4. Time reports (R12.1), burn-down (R13.1), and capacity planning (R14.1) would support this flow directly — these requirements are currently `planned`.

**Gap:** Until R12–R14 are implemented, billings-oriented users must post-process the raw export.
