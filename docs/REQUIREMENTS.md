72|## Use cases
73|
74|Three primary use cases are recorded in the Scooter KG (`has_use_case` facts) and corroborated by `juggler/docs/PROJECT-BRIEF.md §Users`.
75|
76|### UC-1 — Individual contributor: track tasks and plan sprints
77|
78|**Actor:** Developer or knowledge worker managing their own workload.
79|
80|**Goal:** Schedule personal tasks efficiently across working days, track progress, and avoid overcommitment.
81|
82|**Flow:**
83|1. User creates tasks (R1) with duration estimates, priorities, and optional deadlines.
84|2. User organizes tasks into projects (R4), optionally color-coded (R5).
85|3. User runs the scheduler (R11) — tasks are placed in optimal slots respecting constraints.
86|4. User views the calendar grid (R8) to see today's plan alongside calendar events.
87|5. User marks tasks `wip` as they start work, updating `time_remaining` (R6).
88|6. User marks tasks `done` via status update (R2).
89|7. User can invoke AI natural-language commands (R15) to bulk-create or reschedule tasks.
90|8. User configures recurring tasks (R18), splits (R19), locations (R20), and schedule templates (R21) to fine-tune scheduling.
91|
92|**MCP variant:** Same flow via `create_task` / `run_schedule` / `set_task_status` MCP tools (R17).
93|
94|### UC-2 — Team lead: view workload and monitor progress
95|
96|**Actor:** Engineering manager or team lead overseeing a small team.
97|
98|**Goal:** See the overall task backlog, understand scheduling status, and identify blockers.
99|
100|**Flow:**
101|1. Lead uses the MCP interface (R17) or the UI to query tasks by project or status.
102|2. Lead reviews the dependency graph (R10) to identify blocked chains.
103|3. Lead reassigns tasks via task update (R2), adjusting priorities and deadlines.
104|4. Lead uses calendar sync (R7) to ensure team events are reflected in the schedule.
105|
106|**Gap:** Dedicated team-workload view and assignment is not implemented as of 2026-06-15; the above flow is achievable via individual-user flows only.
107|
108|### UC-3 — Freelancer: billable hours and project estimation
109|
110|**Actor:** Independent contractor billing clients per hour or per project.
111|
112|**Goal:** Track time spent on client projects, estimate future work, and produce billing data.
113|
114|**Flow:**
115|1. Freelancer creates tasks (R1) per client project with duration estimates.
116|2. Freelancer marks tasks as `wip` (R6) to record time-in-progress against estimates.
117|3. Freelancer exports data (R22) for external billing use.
118|4. Time reports (R12), burn-down (R13), and capacity planning (R14) would support this flow directly — these requirements are currently `planned`.
119|
120|**Gap:** Until R12–R14 are implemented, billings-oriented users must post-process the raw export.
